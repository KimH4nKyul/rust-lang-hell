# DAY 0: 첫걸음

첫 훈련을 시작한다. 모든 위대한 여정의 시작은 미약하지만, 그 끝은 창대할 것이다. 오늘은 기본기를 다지는 데 집중한다.

### [오늘의 목표]

- [ ] Rust 프로젝트 환경 설정 및 구조 이해
- [ ] 테스트 주도 개발(TDD) 기본 흐름 체험
- [ ] `cargo` 기본 명령어(fmt, clippy, test) 숙달

### [오늘의 과제]

1.  **프로젝트 생성**
    *   `cargo new adder` 명령어로 `adder` 라이브러리 프로젝트를 생성한다. (이 단계는 이미 제가 `task`라는 이름으로 생성할 것이므로 건너뛰고, 생성된 프로젝트 내부에서 다음 단계를 진행하세요.)

2.  **테스트 우선 작성 (TDD)**
    *   `src/lib.rs` 파일에, 아직 존재하지 않는 `add` 함수에 대한 테스트를 먼저 작성한다.
    *   테스트 케이스:
        *   기본 `it_works` 테스트는 `2 + 2 = 4`를 검증하도록 수정한다.
        *   음수 덧셈에 대한 테스트 케이스를 추가한다. (`-2 + -2 = -4`)
        *   0을 더하는 경우에 대한 테스트 케이스를 추가한다. (`5 + 0 = 5`)

3.  **구현**
    *   테스트를 통과하는 `add(a: i32, b: i32) -> i32` 함수를 구현한다.

4.  **검증**
    *   `cargo fmt --check`: 코드 포맷이 표준을 따르는지 확인한다.
    *   `cargo clippy -D warnings`: 잠재적 오류나 비효율적 코드가 없는지 린트 검사를 수행한다. 모든 경고를 에러로 처리한다.
    *   `cargo test`: 모든 테스트가 성공하는지 확인한다.

### [메타인지 질문]

- 왜 기능을 구현하기 전에 테스트부터 작성해야 하는가? TDD의 장점은 무엇인가?
- `cargo clippy`가 지적하는 "경고(warning)"를 무시하지 않고 반드시 수정해야 하는 이유는 무엇일까?
- `add` 함수에서 `i32` 대신 `u8`을 사용했다면 어떤 잠재적 문제가 발생할 수 있는가? (오버플로우)

### [STAR 요약]

- **S**: Rust 개발 환경이 처음 설정됨.
- **T**: TDD 원칙에 따라 간단한 덧셈 함수를 구현하고, 표준 개발 도구(fmt, clippy, test)를 사용해 검증하는 과제를 받음.
- **A**: `cargo new`로 프로젝트를 생성하고, `add` 함수에 대한 테스트를 먼저 작성한 뒤, 이를 통과하는 실제 코드를 구현. `fmt`, `clippy`, `test` 명령어로 코드 품질을 검증.
- **R**: TDD의 기본 사이클을 경험하고, 깨끗하고 테스트된 코드를 작성하는 습관의 첫 단추를 끼움.
